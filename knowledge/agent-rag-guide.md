# AI 秘书 RAG 总说明书

## 文档用途

本文档是给 AI 秘书 agent 使用的总操作手册，用于指导 agent 在不同任务下判断：

1. 是否需要先调用 `retrieve_rag` 查询说明书
2. 查询说明书后应该调用哪些 MCP 工具执行
3. 哪些任务不能只依赖 RAG，必须使用真实工具获取结果

本文档不是面向最终用户的介绍文案，而是 agent 的内部执行规范。

本文档适合作为单文件上传到阿里云百炼 RAG。

## 系统总原则

### 1. RAG 的职责

`RAG` 只存放操作说明书、流程规则、任务处理原则、执行模板和边界约束。

`RAG` 不负责：

- 提供实时数据库结果
- 提供实时文件内容
- 提供实时系统状态
- 代替命令执行
- 代替工具返回值

### 2. MemoryOS 的职责

`MemoryOS` 负责记忆，包括短期、中期、长期记忆。

记忆用于帮助 agent 理解用户偏好、长期背景和已形成的稳定信息，但不能替代当前任务的真实查询。

### 3. thread_store 的职责

`thread_store` 负责前端聊天记录恢复、会话列表展示和历史消息回显。

本地聊天记录不等于模型执行依据。聊天记录主要服务前端恢复和查看，不应被当作实时事实来源。

### 4. Agent 的职责

agent 负责：

- 理解用户意图
- 判断是否需要先查 RAG
- 判断应该调用哪个工具
- 在多个工具之间安排执行顺序
- 总结结果并回复用户

### 5. MCP tools 的职责

MCP tools 负责真实执行，例如：

- 运行命令
- 控制终端
- 查询 RAG
- 修改前端白名单区域
- 获取当前时间

如果任务需要真实结果，必须通过工具执行，而不是只依赖模型推测。

## 任务路由总规则

agent 在开始任务前，先做如下判断：

### A. 如果用户在问“怎么做 / 步骤是什么 / 应该按什么流程 / 规则是什么”

先调用 `retrieve_rag`。

适用例子：

- 如何导入课程表
- 修改前端界面应该按什么流程
- 查询某类任务应该怎么做
- 处理数据库写入前要先做什么

### B. 如果用户在问真实状态、真实内容、真实结果

不能只查 RAG，必须调用真实工具。

适用例子：

- 查询数据库中的课程表
- 查看某个文件内容
- 检查服务是否启动
- 查看命令执行结果
- 查询今天系统时间

### C. 能直接用命令行完成的任务，优先用命令行

只要任务可以通过一次性命令直接完成，就优先使用 `run_cli`。

特别适用于：

- 数据库查询
- 表结构查看
- 文件查找
- 文本搜索
- 一次性脚本执行
- 环境检查

不要为了“先找说明”而绕过本来可以直接执行的命令行任务。

### D. 长任务、交互式任务、需要持续观察输出的任务

优先使用 Terminal 工具链，不要只使用 `run_cli`。

适用例子：

- 启动一个持续运行的服务
- 运行需要持续观察输出的脚本
- 运行需要后续继续输入按键的进程
- 需要保持终端上下文的任务

### E. 时间相关问题

优先使用 `get_current_datetime`。

适用例子：

- 现在几点
- 今天几号
- 今天星期几
- 明天是周几
- 本周如何判断

不能凭模型记忆猜测当前时间。

### F. 前端界面修改

前端界面修改任务优先走前端白名单工具，不允许跳过读取和构建校验。

推荐顺序：

1. 必要时先 `retrieve_rag`
2. `list_frontend_regions` 或直接 `read_frontend_region`
3. `update_frontend_region`
4. `validate_frontend_region_change`

### G. 当任务既涉及规则又涉及真实执行

先查 RAG，再执行真实工具。

适用例子：

- 导入课程表
- 修改学习区或生活区界面
- 需要遵守项目内部流程的数据库更新

## 工具总览

## `run_cli`

用途：

- 运行一次性命令行命令
- 适合数据库查询、表结构查看、文件查询、轻量脚本

特点：

- 适合快速得到一次性结果
- 不适合需要长期保持上下文的任务

限制：

- 禁止 `rm`

适合场景：

- `SHOW TABLES;`
- `DESCRIBE course_schedule;`
- `SELECT ...`
- `ls`
- `cat`
- `rg`
- 一次性脚本运行

不适合场景：

- 需要持续运行的服务
- 需要不断查看新增输出的终端任务
- 需要继续发送按键的进程

## `close_terminal_if_open`

用途：

- 如果 Terminal 已经打开则关闭

适合场景：

- 在重新整理终端环境前清理旧终端

## `open_new_terminal`

用途：

- 打开新的终端窗口

适合场景：

- 开启一个需要长期运行的任务
- 准备执行交互式命令

## `get_terminal_full_text`

用途：

- 获取当前终端窗口完整文本

适合场景：

- 查看长期运行任务的输出
- 读取服务日志和错误信息

## `run_script_in_exist_terminal`

用途：

- 在已存在终端窗口中运行脚本或命令

适合场景：

- 已经打开终端后继续执行命令
- 需要保持同一终端上下文

## `send_terminal_keyboard_key`

用途：

- 向已存在终端窗口发送键盘按键

适合场景：

- 交互式任务
- 需要回车、方向键或字母输入的终端操作

## `retrieve_rag`

用途：

- 查询知识库中的说明书、规则、步骤和操作手册

适合场景：

- 判断某类任务应该按什么流程做
- 查询项目约定和说明
- 查询导入、更新、前端修改等规范

限制：

- 不能把返回内容当作实时数据库结果
- 不能把返回内容当作真实系统状态
- 不能把返回内容当作命令已经执行成功

## `list_frontend_regions`

用途：

- 列出允许修改的前端白名单区域

当前 region id：

- `sidebar-nav`
- `today-plan-panel`
- `study-panel`
- `life-panel`

## `read_frontend_region`

用途：

- 读取指定前端区域源码、文件路径和约束

要求：

- 修改前端前必须先读取当前区域

## `update_frontend_region`

用途：

- 更新指定前端白名单区域源码

要求：

- 只能修改白名单区域
- 必须保留该区域要求的导出
- 修改后要进行校验

## `validate_frontend_region_change`

用途：

- 校验前端区域修改后项目是否能正常构建

要求：

- 修改前端后必须执行

## `get_current_datetime`

用途：

- 获取当前日期、时间、星期和时区

默认时区：

- `Asia/Shanghai`

适合场景：

- 现在几点
- 今天几号
- 今天星期几
- 明天是周几
- 本周判断

## 按任务场景的处理说明

### 场景 1：时间与日期类问题

适用问题：

- 现在几点
- 今天几号
- 今天星期几
- 明天是周几
- 本周安排

处理规则：

1. 直接优先使用 `get_current_datetime`
2. 如果系统已提供当前时间上下文，且足够回答问题，可以直接回答
3. 当任务需要精确时间时，优先工具结果

不要这样做：

- 不要凭模型记忆猜测今天的日期
- 不要把旧上下文当成当前时间

### 场景 2：数据库与命令行类任务

适用问题：

- 查数据库内容
- 查课程表数据
- 看表结构
- 查文件
- 查目录
- 查日志文件

处理规则：

1. 如果用户是在问“如何操作数据库”，先查 `retrieve_rag`
2. 如果用户是在要真实数据库结果，直接优先 `run_cli`
3. 优先使用一次性命令获取真实结果
4. 读取命令输出后再回复

推荐思路：

- 查询数据：`run_cli`
- 查看表结构：`run_cli`
- 查询文件：`run_cli`

不要这样做：

- 不要只查 RAG 然后假装数据库里有结果
- 不要把操作说明书当成真实查询结果

### 场景 3：长任务与交互式终端任务

适用问题：

- 启动后端服务
- 运行长期脚本
- 跑一个会持续输出的程序
- 需要持续读取终端内容

处理规则：

1. 先准备终端环境
2. 打开终端：`open_new_terminal`
3. 在终端中运行命令：`run_script_in_exist_terminal`
4. 读取输出：`get_terminal_full_text`
5. 必要时发送按键：`send_terminal_keyboard_key`

不要这样做：

- 不要只用 `run_cli` 去跑一个需要持续观察的任务
- 不要在需要交互输入时只返回“命令已执行”

### 场景 4：前端界面修改任务

适用问题：

- 修改学习区界面
- 修改生活区界面
- 修改今日计划区
- 修改导航栏

处理规则：

1. 如果用户是在问“应该怎么改前端”或“按什么流程改”，先 `retrieve_rag`
2. 明确目标 region
3. 先调用 `read_frontend_region`
4. 再调用 `update_frontend_region`
5. 最后调用 `validate_frontend_region_change`

当前前端区域：

- `sidebar-nav`
- `today-plan-panel`
- `study-panel`
- `life-panel`

额外要求：

- 不能修改白名单之外的区域
- 不能跳过读取当前区域源码
- 不能跳过构建校验
- 未校验前不能宣称“已经更新成功”

### 场景 5：课程表导入任务

适用问题：

- 用户上传课程表图片、PDF、文本
- 用户要求创建课程信息
- 用户要求导入或更新课程表
- 用户要求写入 `course_schedule`

处理规则：

1. 先判断用户是否明确要求写入数据库
2. 如果没有明确要求，则先确认
3. 若任务涉及具体入库流程，先 `retrieve_rag`
4. 查数据库现状与表结构，优先 `run_cli`
5. 如果已有旧数据，先确认更新策略
6. 读取图片或文本，整理结构化课程信息
7. 按真实 schema 生成 SQL
8. 执行 SQL
9. 写后校验结果

具体要求：

- 未确认前不要写库
- 先查 `course_schedule` 是否存在
- 先查表结构再写 SQL
- 已有旧数据时必须确认覆盖、追加或局部更新策略
- 不要边识别边直接拼 SQL
- 写入后必须校验

推荐数据库检查命令：

```sql
SHOW TABLES;
```

```sql
DESCRIBE course_schedule;
```

```sql
SHOW CREATE TABLE course_schedule;
```

```sql
SELECT COUNT(*) AS total FROM course_schedule;
```

推荐写后校验命令：

```sql
SELECT academic_year, term, weekday, start_time, end_time, course_name, location
FROM course_schedule
ORDER BY weekday, start_time
LIMIT 10;
```

如果识别结果存在歧义：

- 地点不清晰
- 周次不明确
- 教师信息残缺
- 节次没有精确时间

则应先向用户确认，而不是强行入库。

### 场景 6：先问规则再执行的任务

适用问题：

- “我想做某件事，你先告诉我怎么做”
- “这个项目里某类任务的处理流程是什么”
- “你应该如何帮我完成某项导入或更新”

处理规则：

1. 先调用 `retrieve_rag`
2. 读到规则后，再判断是否需要调用其他工具
3. 不要跳过规则说明直接执行

## 禁止行为与边界

以下行为禁止：

### 1. 不要把 RAG 当实时事实来源

RAG 只能告诉 agent“应该怎么做”，不能告诉 agent“现在真实是什么”。

### 2. 不要在未验证前宣称已经成功

适用例子：

- 前端还没 build 通过
- SQL 还没校验
- 服务还没确认启动

### 3. 不要修改前端前跳过 `read_frontend_region`

前端修改必须先读取目标区域源码和约束。

### 4. 不要跳过前端构建校验

前端更新后必须执行 `validate_frontend_region_change`。

### 5. 不要在时间问题上凭模型记忆猜测

时间问题优先 `get_current_datetime`。

### 6. 不要在适合 `run_cli` 的任务上绕远路

如果数据库查询、文件查询、表结构查看可以直接命令行完成，就直接命令行。

### 7. 不要在需要持续观察输出时只用 `run_cli`

长任务和交互任务要使用 Terminal 工具链。

### 8. 不要在用户未确认时执行破坏性数据库操作

尤其是：

- 覆盖已有课表
- 删除旧数据
- 做大范围清空

## 失败时的降级策略

### 1. RAG 查询不到合适说明

处理方式：

- 明确告知未检索到足够说明
- 基于当前工具能力采用最保守的执行方式
- 如果涉及高风险写操作，宁可先确认用户，也不要擅自执行

### 2. `run_cli` 执行失败

处理方式：

- 读取错误输出
- 根据错误信息判断是命令错误、环境问题、权限问题还是数据问题
- 不要假装已经执行成功

### 3. Terminal 工具链执行失败

处理方式：

- 优先读取终端文本
- 必要时重新打开终端并重试
- 如果任务已经进入不明确状态，先向用户说明现状

### 4. 前端构建校验失败

处理方式：

- 依赖前端工具自动回滚
- 向用户说明失败原因
- 不要说“界面已更新”

### 5. 数据库写入后校验失败

处理方式：

- 告知用户写入结果不可靠
- 展示校验异常点
- 不把这次写入当作成功完成

## 推荐执行模板

### 模板 1：查询数据库内容

适用任务：

- 查课程表
- 查表结构
- 查数据库记录

推荐顺序：

1. 判断是否只是问规则
2. 如果不是规则问题，直接 `run_cli`
3. 读取结果
4. 总结回复

### 模板 2：修改学习区或生活区前端

适用任务：

- 修改 `study-panel`
- 修改 `life-panel`
- 修改 `today-plan-panel`
- 修改 `sidebar-nav`

推荐顺序：

1. 如需流程说明，先 `retrieve_rag`
2. `read_frontend_region`
3. `update_frontend_region`
4. `validate_frontend_region_change`
5. 回复结果

### 模板 3：课程表导入

推荐顺序：

1. `retrieve_rag`
2. 确认用户是否真的要写库
3. `run_cli` 查询表结构和现有数据
4. 如果有旧数据，确认更新策略
5. 读取并整理课程表信息
6. 生成 SQL
7. `run_cli` 执行 SQL
8. `run_cli` 做写后校验
9. 总结结果

### 模板 4：时间问题

推荐顺序：

1. `get_current_datetime`
2. 根据工具结果回答

### 模板 5：长任务或交互式任务

推荐顺序：

1. `open_new_terminal`
2. `run_script_in_exist_terminal`
3. `get_terminal_full_text`
4. 必要时 `send_terminal_keyboard_key`
5. 根据终端输出回复

## 最终执行准则

agent 在处理任务时，优先遵循以下顺序：

1. 先判断任务是“要规则”还是“要真实执行”
2. 需要规则时先查 `retrieve_rag`
3. 需要真实结果时直接调用真实工具
4. 能直接用命令行完成的任务优先 `run_cli`
5. 长任务和交互任务走 Terminal 工具链
6. 前端修改任务走前端白名单工具链
7. 时间问题走 `get_current_datetime`
8. 所有高风险写操作都先确认、再执行、再校验

这份文档的目标不是让 agent 背诵规则，而是让 agent 在需要时知道：

- 什么时候该查说明书
- 查完后该用什么工具
- 哪些事不能靠推测
- 哪些执行必须做校验
