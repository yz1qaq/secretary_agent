# 课程表导入与更新知识文档

## 文档用途

本文档用于指导外部代理处理“课程表导入数据库”相关任务。

适用场景包括：

- 用户上传课程表截图、照片、PDF、OCR 文本或表格文本
- 用户要求创建课程信息
- 用户要求导入课表到数据库
- 用户要求更新已有课表
- 用户要求把课程安排写入 `course_schedule` 或类似课程表数据表

本文档不是用户可见的话术模板，而是代理执行任务时应遵循的业务规则与处理流程。

## 核心目标

当用户提供课程表信息并明确要求入库时，代理需要：

1. 确认用户是否真的需要创建或更新课程信息
2. 检查数据库中是否已有课程表，以及表结构是否匹配
3. 读取并整理课程表内容为结构化数据
4. 生成并执行安全的 SQL
5. 校验写入结果是否成功

## 处理原则

### 1. 未确认前不要写库

如果用户上传了课程表，但没有明确说“导入”“创建”“写入数据库”“更新课表”，代理不能直接执行数据库写入。

此时应先询问用户是否需要秘书创建课程信息。

推荐问法：

`我可以帮你把这份课程表整理并写入数据库。你希望我现在创建或导入课程信息吗？`

### 2. 先看表结构，再写 SQL

代理不能假设数据库表结构固定。执行写入前，必须先确认：

- 课程表对应的数据表是否存在
- 表名是否确实是 `course_schedule`
- 字段名、字段类型、主键、唯一键是否与当前任务兼容

常用检查方式：

```sql
SHOW TABLES;
```

```sql
DESCRIBE course_schedule;
```

或：

```sql
SHOW CREATE TABLE course_schedule;
```

### 3. 有旧数据时必须先确认更新策略

如果数据库中已经存在课程表数据，不能直接覆盖。

需要先询问用户希望采用哪种策略：

- 覆盖原有课表
- 追加新课表
- 只更新某个学年/学期
- 只更新当前导入涉及的课程

推荐问法：

`当前数据库中已经存在课程表数据。你希望我覆盖原有课表、追加新课表，还是只更新某个学期的数据？`

如果用户没有明确答复，不要进行删除型操作。

### 4. 先结构化，再生成 SQL

代理不应一边读取课程表附件，一边直接拼接 SQL。

正确顺序是：

1. 读取图片、PDF 或文本
2. 提取课程字段
3. 整理成结构化记录
4. 基于真实 schema 生成 SQL

### 5. 写入后必须校验

代理不能把“执行了 SQL”当作“导入成功”。

写入后必须至少执行以下校验：

- 统计当前表行数
- 查看刚写入的部分记录
- 检查关键字段是否为空
- 检查时间、星期、周次格式是否明显异常

## 推荐处理流程

### Step 1. 确认用户意图

如果用户仅上传课程表附件，但未明确要求写库：

- 先询问是否需要创建课程信息
- 若用户未确认，则只做读取、总结或展示，不写数据库

### Step 2. 检查数据库现状

在用户确认需要导入后：

1. 检查课程表相关表是否存在
2. 检查表结构
3. 检查是否已有数据

例如：

```sql
SELECT COUNT(*) AS total FROM course_schedule;
```

### Step 3. 读取并整理课程表内容

代理需要从用户上传的图片、PDF 或文本中提取课程信息。

理想情况下，每条课程记录应尽量包含以下字段：

- 学年
- 学期
- 星期
- 开始时间
- 结束时间
- 课程名
- 地点
- 教师
- 周次
- 原始来源摘要 `raw_source`

### Step 4. 处理信息不完整或有歧义的情况

如果课程表识别结果不完整，代理应先确认，而不是直接入库。

常见歧义包括：

- 时间只有“第 3-4 节”，没有具体时刻
- 地点识别模糊
- 教师姓名不完整
- 周次不清楚
- 多行文本合并错位

推荐问法：

`我已经识别出大部分课程信息，但其中有几条记录的地点或周次不太确定。要我先展示识别结果供你确认，再继续写入数据库吗？`

### Step 5. 根据用户选择生成 SQL

根据用户确认的策略生成 SQL：

- 覆盖：通常是先删除明确范围内的旧数据，再插入新数据
- 追加：直接插入，但应考虑重复数据
- 局部更新：只更新指定学期、学年或指定课程范围

如果表存在唯一键或可判重字段，优先考虑 upsert。

### Step 6. 执行 SQL

执行 SQL 时遵循保守策略：

- 删除必须有明确范围
- 不允许在未确认情况下做全表清空
- 如果支持事务，优先事务执行

推荐执行顺序：

1. 开始事务
2. 执行删除或更新
3. 执行插入
4. 做校验查询
5. 检查无误后提交

### Step 7. 校验结果

写入完成后，至少执行：

```sql
SELECT COUNT(*) AS total FROM course_schedule;
```

```sql
SELECT academic_year, term, weekday, start_time, end_time, course_name, location
FROM course_schedule
ORDER BY weekday, start_time
LIMIT 10;
```

代理需要确认：

- 数据条数合理
- 课程名正常
- 星期和时间字段格式正确
- 地点、教师、周次没有大面积空值

## 推荐字段映射

如果数据库表结构允许，课程记录建议包含如下信息：

| 字段含义 | 建议字段名 |
| --- | --- |
| 学年 | `academic_year` |
| 学期 | `term` |
| 星期 | `weekday` |
| 开始时间 | `start_time` |
| 结束时间 | `end_time` |
| 课程名 | `course_name` |
| 地点 | `location` |
| 教师 | `teacher` |
| 周次文本 | `week_text` |
| 周次范围 | `week_range` |
| 原始识别来源 | `raw_source` |

注意：实际写库前仍应以真实数据库 schema 为准。

## 常见错误与避免方式

### 错误 1：用户未确认就直接写库

避免方式：

- 只要用户没有明确表达“创建/导入/更新课程信息”，就先确认

### 错误 2：不检查 schema 就直接写 SQL

避免方式：

- 先执行 `DESCRIBE course_schedule` 或 `SHOW CREATE TABLE`

### 错误 3：发现旧数据后直接覆盖

避免方式：

- 必须让用户明确选择覆盖、追加或局部更新

### 错误 4：OCR 结果不稳定仍强制入库

避免方式：

- 对低置信度字段先展示预览并询问用户

### 错误 5：执行成功就认为导入成功

避免方式：

- 执行写后校验，查看样例记录

## 推荐输出格式

当代理完成课程表导入任务时，输出应尽量包含：

- 用户目标理解
- 数据库检查结果
- 是否存在旧数据
- 用户确认的更新策略
- 识别出的课程条数
- SQL 执行结果
- 写后校验结果
- 是否存在不确定项
